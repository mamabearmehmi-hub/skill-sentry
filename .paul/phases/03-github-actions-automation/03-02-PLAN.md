<objective>
## Goal
Create the daily scraper script that discovers new MCP repos via GitHub API topic search, and the scheduled GitHub Action that runs it on a cron. Each discovered repo is audited via the reusable-auditor workflow and committed to the registry — building the database automatically.

## Purpose
The scraper is the growth engine. Without it, the registry only grows when users submit repos manually. With the daily scraper, Skill Sentry automatically discovers and audits new MCP skills/servers, building credibility and coverage without manual effort. This is half of the "virtuous cycle" (scraper + submit button).

## Output
- `scripts/scraper.ts` — discovers repos by GitHub API topic search, filters already-audited, outputs URLs
- `.github/workflows/scheduled-scrape.yml` — daily cron calling scraper then reusable-auditor for each repo
</objective>

  <name>Task 1: Create scraper.ts — GitHub API topic search + dedup</name>
  <files>scripts/scraper.ts</files>
  <action>
    Create scripts/scraper.ts that:

    1. Reads the current registry.json to build a Set of "owner/name" already audited
    2. For each topic in GITHUB_TOPICS:
       - Calls GitHub REST API: GET /search/repositories?q=topic:{topic}&sort=updated&per_page=30
       - Uses fetch() (built into Node 20+)
       - Passes Authorization header if GH_TOKEN env var is set
       - Parses response, extracts: full_name, html_url, stargazers_count, description, topics
    3. Deduplicates:
       - Skips repos already in registry (by owner/name)
       - Skips repos already seen in this scrape session (cross-topic dedup)
    4. Outputs newline-delimited JSON to stdout, one object per new repo:
       { "url": "https://github.com/...", "owner": "...", "name": "..." }
    5. Error handling:
       - If a topic search fails (403 rate limit, network error): log to stderr, continue
       - If GH_TOKEN not set: warn but proceed (unauthenticated = 60 req/hr)
    6. Exit 0 always (partial results are fine)

    CLI usage:
      GH_TOKEN=xxx npx tsx scripts/scraper.ts

    Import: GITHUB_TOPICS from lib/constants.ts, Registry from lib/types.ts

    Avoid:
    - Do NOT install Octokit — use native fetch()
    - Do NOT audit repos — only discover them (the workflow handles auditing)
    - Do NOT write to registry.json — only read it for dedup
    - Limit to 30 repos per topic (first page only) to stay within rate limits
  </action>
  <verify>npx tsc --noEmit scripts/scraper.ts</verify>
  <done>AC-1 satisfied: Topic search + dedup. AC-2 satisfied: Error handling. AC-4 satisfied: GH_TOKEN env var.</done>

<task type="auto">
  <name>Task 2: Create scheduled-scrape.yml — daily cron workflow</name>
  <files>.github/workflows/scheduled-scrape.yml</files>
  <action>
    Create .github/workflows/scheduled-scrape.yml:

    ```yaml
    name: Daily Scrape

    on:
      schedule:
        - cron: '0 6 * * *'  # Daily at 06:00 UTC
      workflow_dispatch: {}   # Manual trigger for testing

    jobs:
      discover:
        runs-on: ubuntu-latest
        outputs:
          repos: ${{ steps.scrape.outputs.repos }}
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '20'
              cache: 'npm'
          - run: npm ci
          - name: Discover new repos
            id: scrape
            env:
              GH_TOKEN: ${{ secrets.GH_TOKEN }}
            run: |
              REPOS=$(npx tsx scripts/scraper.ts)
              echo "repos<<EOF" >> $GITHUB_OUTPUT
              echo "$REPOS" >> $GITHUB_OUTPUT
              echo "EOF" >> $GITHUB_OUTPUT

      audit:
        needs: discover
        if: needs.discover.outputs.repos != ''
        strategy:
          matrix:
            repo: ${{ fromJson(format('[{0}]', needs.discover.outputs.repos)) }}
          max-parallel: 3
          fail-fast: false
        uses: ./.github/workflows/reusable-auditor.yml
        with:
          repo_url: ${{ matrix.repo.url }}

      commit:
        needs: [discover, audit]
        if: always() && needs.discover.outputs.repos != ''
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '20'
              cache: 'npm'
          - run: npm ci
          - name: Update registry with all results
            env:
              GH_TOKEN: ${{ secrets.GH_TOKEN }}
            run: |
              echo '${{ needs.discover.outputs.repos }}' | while IFS= read -r line; do
                URL=$(echo "$line" | npx tsx -e "
                  let d='';
                  process.stdin.on('data',c=>d+=c);
                  process.stdin.on('end',()=>{try{console.log(JSON.parse(d).url)}catch{}})
                ")
                if [ -n "$URL" ]; then
                  npx tsx scripts/cli.ts "$URL" > /tmp/result.json 2>/dev/null && \
                  npx tsx scripts/update-registry.ts /tmp/result.json || \
                  echo "Failed to audit: $URL"
                fi
              done
          - name: Commit updated registry
            uses: stefanzweifel/git-auto-commit-action@v5
            with:
              commit_message: "scrape: daily discovery $(date -u +%Y-%m-%d)"
              file_pattern: public/data/registry.json
    ```

    Key points:
    - Cron at 06:00 UTC daily + workflow_dispatch for manual testing
    - Scraper discovers, then each repo is audited + committed in one batch
    - max-parallel: 3 to respect GitHub API rate limits
    - fail-fast: false so one failed audit doesn't block others
    - GH_TOKEN passed via secrets for authenticated API access
    - Final commit bundles all registry updates into one commit

    NOTE: The matrix-based approach with reusable workflows may have limitations
    in passing outputs back. The commit job uses a simpler approach: re-run
    cli.ts + update-registry.ts for each discovered repo sequentially, then
    commit once. This is more reliable than trying to pass matrix outputs.

    Avoid:
    - Do NOT trigger Vercel redeploy — happens automatically on push
    - Do NOT add notification steps (Slack, email, etc.)
  </action>
